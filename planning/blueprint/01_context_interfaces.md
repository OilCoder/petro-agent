# Context & Interfaces — petro-agent

## External inputs (shape + source)

### Primary input: LAS files

| Field | Detail |
|---|---|
| Format | LAS 2.0 (CWLS standard) |
| Loader | lasio — produces a `lasio.LASFile`; curves exposed as numpy arrays |
| Depth index | 1-D float array (MD, metres or feet as declared in `~Well` header) |
| Curve set — required | `GR` (gamma ray, API), `RT` (true resistivity, Ω·m), and at least one of `RHOB` (bulk density, g/cc) or `NPHI` (neutron porosity, v/v or p.u.) — absence of `GR`, `RT`, or both porosity curves rejects the file at intake; a file with only one porosity curve is degraded to the single-porosity PHIE path, not rejected (see `03_source_sink_contracts.md`) |
| Curve set — optional | `CALI` (caliper, inches) and `DCAL` (differential caliper = CALI − bit size, inches), used for bad-hole masking. If `DCAL` is absent, masking uses `CALI` vs. a bit-size constant from the config library; if both are absent, bad-hole masking is unavailable and the affected computations are tier-downgraded with a degradation entry (no file rejection). Also accepted: `DT` (compressional slowness, µs/ft) and `PEF` (photoelectric factor, barns/electron) — reserved for the Phase 3 M-N / lithology cross-plots, not on the v1 quantitative path (canonical schema in `03_source_sink_contracts.md`) |
| Null / absent curves | lasio sets absent samples to a sentinel (default −9999.25); the QC gate (Phase 1) intercepts and masks before any computation |
| LAS header fields consumed | `WELL`, `UWI` (or `API`), `COMP`, `STRT`, `STOP`, `STEP`, `NULL`, `PROV` (formation/region tag used to select the Larionov variant) |
| Units assumed at intake | RHOB in g/cc (range check 1.0–3.0); NPHI in v/v fraction (range check 0.0–1.0); GR in API; RT in Ω·m; CALI in inches |
| Development dataset | Kansas / Schaben wells (Paleozoic — Larionov old-rocks variant; zero acquisition friction; quality known) |
| Regression / benchmark dataset | VOLVE (Equinor open dataset, North Sea); carries an accepted petrophysical interpretation used for pipeline validation in Phase 8 |
| Data location | `data/` directory — gitignored, not bundled with the repository; each run references files by path supplied at invocation |

### Configuration inputs

| Input | Format | Source |
|---|---|---|
| Regional / well-specific parameters | JSON config file (versioned, hash-pinned at run time) | Config library in `src/params/` — human-authored offline |
| Provenance tier for each parameter | Enum field in the same JSON (`core` \| `offset` \| `default`) | Set by the expert who builds the config; not determined by an agent at runtime |
| Validator harness version | Declared in config; verified by hash at pipeline start | Fixed artefact under version control |

## Outputs / artifacts

### Prose report

- Format: Markdown (primary); convertible to HTML or PDF via an external renderer — the pipeline does not bundle a renderer.
- Structure: per-zone sections (zone header, Vsh/PHIE/Sw summary, cutoff rationale, net pay, confidence tier, limitations stated explicitly).
- Tone is controlled by the confidence tier of each block (firm / qualified / explicitly-bracketed); the writer agent has no discretion to override the tier.
- Language: English.

### Field-scale rollup (whole-field scope)

- v1 is **field-scale**: the pipeline emits per-well report blocks PLUS a field-rollup layer over all Schaben wells.
- **Field rollup summary**: aggregate net pay / NTG / HCPV across the field plus a per-well summary table, produced by a deterministic aggregation pass over the per-well ledger entries (no LLM computation; schema in `03_source_sink_contracts.md`).
- **Field net-pay / quality map**: a matplotlib map of net pay / reservoir quality across well locations, saved to `outputs/` (gitignored).
- **Cross-well zone-correlation panel**: a multi-well panel aligning zonation across wells, saved to `outputs/` (gitignored).

### JSON ledger

- One ledger file per run, emitted to `outputs/` (gitignored).
- Per-computation schema (canonical definition in `03_source_sink_contracts.md`): `{"depth_range_m": [top, base], "output_curve": "<vsh|phie|sw>", "input_curves": [...], "function": "<name>", "function_version": "<semver>", "function_module": "<dotted path>", "parameters": {...}, "parameter_provenance": {...}, "result_p50": <number>, "validator_ids": [...]}`.
- Every number in the prose report must resolve to a ledger entry via its `(depth_range_m, output_curve)` key — O(1) lookup.
- Ledger completeness is a gating condition: the pipeline does not emit the report until the ledger covers all quantitative claims.

### Cross-plots (PNG)

- Neutron-density and M-N crossplots generated by matplotlib.
- Saved to `outputs/` (gitignored).
- Embedded by reference in the report; used as evidence for lithology and model-mismatch flags.
- Not interactive; static image files.

### Run metadata (within the ledger)

- Pinned library versions: `lasio`, `numpy`, `langgraph`, `ollama` client.
- Pinned model identifiers: Qwen3:30b-a3b (full model tag as returned by the Ollama registry), Llama3.1:8b (if used as adversarial reviewer).
- Config hash (SHA-256 of the parameter JSON file).
- Random seeds for any stochastic step (relevant in Phase 7 when Monte Carlo or multi-seed robustness is active).
- Wall-clock timestamp and git commit SHA of the `src/` tree.

## Integration points

### lasio (LAS I/O)

- Invoked at pipeline entry in `src/io/` to load the LAS file.
- Returns a `lasio.LASFile`; the loader extracts the curve arrays and header fields into an internal typed structure before handing off to the QC gate.
- Version is pinned in `pyproject.toml`; the pin is logged in the ledger.
- Known gotcha: lasio crashes when `~Other` appears before `~Curve` in the LAS header — the loader guards against this by re-ordering sections before parse.

### numpy (numerics)

- All deterministic petrophysical computation (Larionov→Vsh, density-neutron→PHIE, Archie→Sw) operates on numpy arrays in `src/petrophysics/`.
- Physical-bounds validators operate on numpy arrays in `src/validators/`.
- No pandas dependency on the quantitative path — numpy arrays indexed by depth position.
- Version is pinned; the pin is logged in the ledger.

### LangGraph (orchestration)

- Implements the deterministic state machine in `src/orchestrator/`.
- Owns the compute→validate→correct loop, the gating compuertas, and the termination condition.
- LangGraph nodes call agents; LangGraph edges and conditions are deterministic Python — no LLM decides the transition.
- The circuit breaker (non-convergence guard) is a deterministic counter inside LangGraph: if the correctable-objection count does not decrease monotonically, the loop halts and marks the run as "did not converge".

### Ollama (local LLM inference)

- All LLM calls go through the Ollama HTTP API on `localhost` (default port 11434).
- **Primary model**: Qwen3:30b-a3b — used for the compute agent (method/parameter selection), writer (prose), and claim verifier.
- **Secondary model**: Llama3.1:8b — used for fast iteration during development; the adversarial reviewer (Phase 6) — decision (a) CLOSED 2026-06-25, second model family.
- Models must be pre-pulled to the Ollama instance before a run; the pipeline does not pull models at runtime.
- Both models run locally; no cloud LLM endpoint is called at any point during report generation.
- Model tags are pinned at invocation and logged in the ledger.

### pytest (golden tests)

- Golden tests in `tests/` verify the vetted petrophysical engine; they are not called by the runtime pipeline — they are a pre-deploy gate run by the developer.
- A failing golden test must block deployment; the pipeline assumes the engine is correct if all tests pass.

### matplotlib (cross-plots)

- Used in `src/validators/` to generate neutron-density and M-N crossplots.
- Output is a PNG saved to `outputs/`; matplotlib is not in the critical path for the quantitative result.

## Environment assumptions

| Assumption | Value / constraint |
|---|---|
| OS | Linux on WSL2 (Windows host); devcontainer / Docker is the target reproducibility envelope |
| Python version | Declared in `pyproject.toml`; consistent across host WSL and devcontainer |
| GPU | Approximately 16 GB VRAM (WSL + GPU passthrough); Qwen3:30b-a3b must be loaded quantized (Q4\_K\_M or equivalent) to fit within this ceiling |
| Ollama service | Running locally before pipeline invocation; models pre-pulled; accessible at `localhost:11434` |
| File system | `data/` and `outputs/` are local directories, gitignored; paths are supplied at invocation, not hardcoded |
| Network | No outbound network calls during a report run; all inference is local |
| Concurrency | Single-run serial execution; no parallel report generation assumed in v1 |
| Data files | Not bundled with the repository; operator places LAS files in `data/` before invoking the pipeline |

## Out-of-scope interfaces

- **Cloud LLM APIs** (OpenAI, Anthropic, Cohere, etc.) — not called at any point during report generation. This is an Invariant in the Charter.
- **Streaming or real-time log data** — the system processes static LAS files only; acquisition-time or WITSML streaming is excluded.
- **Interactive UI** (Streamlit, Gradio, web front-end) — the pipeline runs headless, invoked from the command line or the promptloop script; no UI surface is provided in v1.
- **Third-party petrophysical libraries for the core equation layer** (e.g., welly, striplog, petrolib) — the Larionov, density-neutron PHIE, and Archie equations are authored in-project for full ledger transparency. This is a Non-goal in the Charter.
- **Database or cloud storage backends** — inputs are local files; outputs write to local `outputs/` only.
- **LAS 3.0 or DLIS formats** — only LAS 2.0 is supported in v1; other well-log formats are out of scope.
- **Human-in-the-loop callbacks or approval gates** — per-report human interaction is the prior MVP scope (agente-llm-registros); it is not part of this system.
- **Automatic model pulling or version resolution** — models must be pre-pulled to Ollama; the pipeline does not manage model lifecycle.
- **RAG over petrophysical papers** — retrieval-augmented generation for parameter justification is excluded from v1 (decision (d) RESOLVED 2026-06-24 = no RAG). A static curated citations table supplies parameter provenance instead (see Charter and `03_source_sink_contracts.md`); it is not a v1 interface.

## Open questions

- **LAS unit variants at intake** (resolved): NPHI is sometimes delivered in percent (0–100) rather than v/v fraction (0.0–1.0), and RHOB occasionally appears in kg/m³ (1000–3000) rather than g/cc (1.0–3.0). The policy is fixed in `03_source_sink_contracts.md` (the source of truth for the unit-variant contract): auto-convert when the range is unambiguous (`NPHI_max > 2.0` → `/100`, `RHOB_max > 10.0` → `/1000`, each logged as a `unit_conversion` edit) and reject with a diagnostic when the range is ambiguous (e.g. NPHI between 1.0 and 2.0). Applied at the `load` stage (`04_pipeline_architecture.md` Stage 1).
- **VOLVE curve name mapping** (location fixed): VOLVE uses Equinor-internal curve mnemonics that may differ from the Kansas/Schaben defaults (e.g., `RHOZ` vs. `RHOB`, `NPOR` vs. `NPHI`). The mnemonic-alias table lives at `src/params/mnemonic_aliases.json` (canonical/alias table in `03_source_sink_contracts.md`) and is applied at the `load` stage, first match wins (`04_pipeline_architecture.md` Stage 1 step 2). The remaining Phase 8 work is to add the confirmed VOLVE aliases to that table, not to decide where it lives.
- **Config JSON schema version**: the exact JSON schema for the parameter config file (field names, units, provenance enum values) is defined during Phase 2. The ledger's per-entry `parameters` field format follows the same schema; finalizing the schema before Phase 2 is a prerequisite for ledger design.
