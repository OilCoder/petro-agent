# Site Style Guide — petro-agent

This document is the style anchor for the GitHub Pages landing site published from
`docs/`. It governs messaging, tone, visual identity, page structure, and the
technical implementation of the public-facing portfolio page. It builds on the
Charter's goal statement and stakeholder list (`00_charter.md`) and is distinct from
the report output format (defined in `03_source_sink_contracts.md`) and the code
documentation layer (generated by `/document` into `documentation/`).

---

## Purpose & audience

The `docs/` site is a single-page portfolio landing that communicates the petro-agent
project to three audiences simultaneously:

| Audience | What they need to see | Success signal |
|---|---|---|
| Recruiters / non-technical evaluators | Evidence the system is trustworthy and production-grade, not just impressive-sounding | They reach out or bookmark the repo |
| Potential clients (petrophysicists, operators) | Correct problem framing, defensible methodology, honest ceiling | They ask for a demo or a conversation |
| Technical peers (ML engineers, petrophysicists, collaborators) | Architectural soundness, ledger design, calibration methodology, correct use of domain knowledge | They fork, cite, or open an issue |

The site is not a tutorial, not a product sales page, and not a research paper.
It is a concise, verifiable argument that the system does what it claims, and that
what it claims is worth doing.

## Messaging / positioning

### The one-sentence pitch

> "An autonomous petrophysical report engine: every number from tested deterministic
> code, every claim traceable to its evidence, and confidence calibrated — not assumed."

This sentence must appear in the hero section verbatim (or as a direct paraphrase).
It encodes the three pillars:

1. **Tested deterministic code** — the LLM never computes; it orchestrates.
2. **Every claim traceable** — the JSON ledger ties each output to input curves,
   function version, parameters, and parameter provenance.
3. **Calibrated confidence** — the system assigns `FIRM`, `QUALIFIED`, or `BRACKETED`
   tiers based on parameter provenance; tone is bound to the tier by policy-as-code.

### The honest ceiling statement

A short paragraph under the pitch must state the honest ceiling explicitly:

> "The system does not promise correct results. It promises that confidence is
> calibrated: when parameters are unconstrained by core data, the output says so
> explicitly and states the propagated uncertainty range. Honest about what it does
> not know — that is the design, not a disclaimer."

This language is non-negotiable. The Charter's first Non-goal forbids promising
"always correct" results. The site must not use language that implies it.

### What the site must NOT say

- "AI-powered petrophysics" (implies the LLM computes the numbers — it does not).
- "Accurate interpretation" or "reliable results" without qualification.
- Any implied claim that the system replaces domain-expert judgment for well evaluation.
- Phrases that suggest the LLM is the primary reliability mechanism (the deterministic
  harness is the primary reliability mechanism; the LLM orchestrates and writes prose).

### Positioning in three bullets

The site may use a short three-bullet "what makes this different" block:

1. The engine is deterministic, version-pinned, and golden-tested — not generated by
   the LLM at runtime.
2. The confidence tier is assigned by a deterministic gating stage, not by the writer
   agent — tone cannot outrun the evidence.
3. Every number traces in a JSON ledger to its input curve, function, version,
   parameters, and validator results.

## Tone & voice

### Governing principle

The voice is that of a careful engineer who solved a hard problem honestly. It does
not boast, does not hedge into vagueness, and does not use adjectives where numbers
or mechanisms can speak.

### Positive constraints

- **Technical precision over marketing language.** Name the mechanism: "the
  LangGraph orchestrator routes the objection to the gating stage" is better than
  "our intelligent pipeline handles errors automatically."
- **Concrete over comparative.** "MAE < 0.03 v/v on PHIE against the VOLVE accepted
  interpretation" is better than "state-of-the-art accuracy."
- **First-person singular, sparing.** Acceptable in the author bio / about section.
  Avoid "we" — this is a solo portfolio project.
- **Active voice throughout.** "The ledger records parameter provenance" not "parameter
  provenance is recorded by the ledger."

### Negative constraints

- No exclamation points.
- No "cutting-edge", "revolutionary", "powerful", "smart", or "robust" as adjectives
  unanchored to a mechanism or measurement.
- No "etc." at the end of lists — complete the list or cut it.
- No teaser language ("coming soon") for features not yet built. Open questions in the
  site's context are listed under `## Open questions` in this document, not on the page.

### Calibration-specific language

When describing the confidence system on the page, use the exact tier names from the
system: `FIRM`, `QUALIFIED`, `BRACKETED`. Do not substitute synonyms (e.g., "high
confidence", "medium confidence") — the tier names are part of the technical interface
and their precise meaning is described in the "calibrated-confidence explainer" section.

## Visual identity

### Palette

| Role | Value | Usage |
|---|---|---|
| Background | `#0d1117` | Page background (GitHub dark tone) |
| Surface | `#161b22` | Cards, code blocks, diagram background |
| Border | `#30363d` | Dividers, card borders |
| Primary text | `#e6edf3` | Body copy |
| Secondary text | `#8b949e` | Captions, metadata, labels |
| Accent — green | `#3fb950` | FIRM tier indicator, positive signals |
| Accent — yellow | `#d29922` | QUALIFIED tier indicator, warnings |
| Accent — red/orange | `#f85149` | BRACKETED tier indicator, limitations |
| Accent — blue | `#58a6ff` | Links, interactive elements |
| Code background | `#1f2428` | Inline and block code |

This palette is drawn from GitHub's dark theme so the page reads as native alongside
the repository. It is not configurable per-section — the palette applies uniformly
to prevent a "marketing site" feel that would contradict the technical tone.

### Typography

| Role | Font | Size (desktop) | Weight |
|---|---|---|---|
| Page title / hero | `'JetBrains Mono', monospace` | 2.4rem | 700 |
| Section headers (`h2`) | `'Inter', sans-serif` | 1.5rem | 600 |
| Body copy | `'Inter', sans-serif` | 1rem | 400 |
| Code (inline + block) | `'JetBrains Mono', monospace` | 0.9rem | 400 |
| Tier badges | `'JetBrains Mono', monospace` | 0.75rem | 700 |
| Caption / metadata | `'Inter', sans-serif` | 0.85rem | 400 |

Both fonts are loaded from Google Fonts (or self-hosted in `docs/fonts/` for offline
reproducibility). No fallback to system fonts for the hero — the monospace treatment
of the title is part of the identity.

### Iconography / imagery

- No stock photography.
- No generic "AI brain" or "network node" illustrations.
- Diagrams are SVG, drawn to match the dark palette. Two diagrams are required on the
  page (see Page structure below): the pipeline architecture and the confidence-tier
  explainer.
- The demo section uses either a screen-recording GIF (preferred, < 5 MB) or a
  static screenshot with a "play" affordance linking to a hosted video. No auto-play.
- Cross-plot PNGs (from `outputs/`) may be embedded as evidence in the traceability
  showcase section. They must carry a caption with the well UWI and the parameter
  values used.

## Page structure (sections)

The page is a single-scrolling HTML document (`docs/index.html`). No navigation bar
with sub-pages; all content is on one page with anchor links in the header. Section
order is fixed.

### 1. Hero + pitch

**Content**:
- Project name: `petro-agent`
- One-sentence pitch (verbatim from Messaging above).
- The honest ceiling statement (one paragraph, max 60 words).
- Two CTAs side by side: `[View on GitHub]` (links to repo) · `[Read the Wiki]`
  (links to `https://github.com/OilCoder/petro-agent/wiki`).

**Constraints**:
- Hero is full viewport height on desktop (100vh), min 480px on mobile.
- No carousel, no animated text, no typing effect.
- Background is page background colour — no gradient, no image.

### 2. The problem

**Content**:
- A concise statement of the problem the system addresses: autonomous petrophysical
  reporting without the two failure modes that make it hard — LLMs that compute numbers
  they hallucinate, and confidence that does not track actual data quality.
- May use a 2-column "old way / this system" comparison table. If used, the left
  column must be honest (it describes a real pain point, not a strawman).
- No more than 200 words total.

### 3. Architecture diagram

**Content**:
- An SVG diagram of the pipeline:
  `load → qc_gate → compute → validate → typify_objections → [correct loop]* → gating → zonate → write → claim_verify → emit`
  (the `zonate` stage delineates net-pay zones and produces `pipeline_net_pay_m`)
- Annotated with the key architectural decision: "deterministic state machine — not
  an LLM" for the orchestrator node.
- The three objection types (`mechanical`, `support`, `irreducible`) shown as
  routing labels on the loop.
- The ledger as a sidebar artefact receiving writes from every stage.

**Constraints**:
- SVG, not a raster image, so it scales on all viewports.
- Colours from the palette. Nodes use the surface colour; edges use secondary text;
  the ledger sidebar uses the blue accent.

### 4. Demo

**Content**:
- A GIF or screenshot of one end-to-end run: LAS file in → prose report + ledger out.
- The GIF shows the terminal output or a rendered report excerpt. It does not show
  hallucinated numbers — it must be a real run on the Kansas/Schaben dataset.
- Caption: well name, Larionov variant used, number of depth samples processed.

**Constraints**:
- GIF: max 5 MB, 800px wide, ≤ 30 seconds. Hosted in `docs/assets/demo.gif`.
- If the demo is a screenshot, a link to the full report output (in `outputs/` or a
  pinned GitHub Gist) must accompany it.
- This section is added after Phase 5 delivers the first end-to-end prose report.
  Before that, the section placeholder reads: "Demo added after Phase 5 (first
  autonomous report)." — a single sentence, no "coming soon" style.

### 5. Ledger / traceability showcase

**Content**:
- A code block showing a representative JSON ledger entry for one computation: the
  `computation` object with `depth_range_m`, `output_curve`, `function`,
  `function_version`, `parameters` (each with `value`, `unit`, `provenance`),
  and `confidence_tier`.
- A one-paragraph prose explanation of what it means: every emitted number in the
  report has an entry like this; a spot-check resolves to its source in O(1) lookup.
- An optional second code block showing the `validators` array entry (claim verifier
  `result = PASS`) for the same computation.

**Constraints**:
- The JSON in the code block must be a real ledger excerpt from an actual run, not an
  invented sample. It is added after Phase 8 (or Phase 5 if the ledger is partially
  populated before full calibration).
- A comment (`// ...` or `# example from run YYYY-MM-DD`) identifies the source run.

### 6. Calibrated-confidence explainer

**Content**:
- A three-row visual table (not a prose paragraph) showing the three confidence tiers.
  The condition column states the base tier from parameter provenance only; irreducible
  objections are described separately below the table as a downgrade step (this mirrors
  the two-step gating mechanism in `04_pipeline_architecture.md` Stage 7 and must not be
  folded back into the condition):

  | Tier (base) | Provenance condition | Writer must say |
  |---|---|---|
  | `FIRM` | All high-leverage parameters from core or offset calibration | Declarative: "Net pay is 12 m." |
  | `QUALIFIED` | At least one parameter is offset-derived | Qualified: "Net pay is approximately 12 m." |
  | `BRACKETED` | Any parameter is default (no local calibration) | Explicitly bounded: "Given no core calibration, Sw ranges 0.28–0.44. Point estimate of 0.35 should not be used without offset calibration." |

- A sentence after the table: "Each irreducible objection (model mismatch, data-limited)
  downgrades the base tier by one level, floored at BRACKETED. The tier is assigned by
  the deterministic gating stage. The writer agent cannot override it."

### 7. Tech stack

**Content**:
A compact table:

| Layer | Technology |
|---|---|
| LAS I/O | lasio |
| Numerics | numpy |
| Petrophysical engine | project-owned (Larionov → Vsh, density-neutron → PHIE, Archie → Sw) — golden-tested |
| Orchestration | LangGraph (deterministic state machine) |
| LLM — primary agent / writer | Qwen3:30b-a3b (local via Ollama) |
| LLM — adversarial reviewer (Phase 6) | Llama3.1:8b or Qwen3:30b-a3b adversarial prompt — local via Ollama; choice deferred to Phase 6 (TBD) |
| Validators | numpy + matplotlib (physical bounds, cross-plots) |
| Traceability | JSON ledger (per-run, version-pinned) |
| Tests | pytest (golden tests on every engine function) |
| Environment | Python on WSL → devcontainer / Docker |

A one-sentence note below the table: "No cloud LLM is called during a report run.
All inference is local via Ollama."

### 8. Links

**Content**:
Four links in a simple grid or inline list:
- GitHub repository: `https://github.com/OilCoder/petro-agent`
- GitHub Wiki (code documentation): `https://github.com/OilCoder/petro-agent/wiki`
- Kansas/Schaben data acknowledgement (link to the public source)
- VOLVE dataset acknowledgement (link to Equinor's open data page)

Author contact (optional): a mailto link or LinkedIn, kept subdued (secondary text
colour, no CTA button).

## Page tech stack

### Recommendation: plain HTML + CSS

The `docs/index.html` is hand-authored HTML with a single embedded or linked CSS
file (`docs/style.css`). No static site generator, no framework, no build step.

**Rationale**:

- Jekyll is GitHub Pages' default, but it requires Ruby, a `Gemfile`, a `_config.yml`,
  and a build pipeline — all overhead for a page that has exactly one file of content.
  Jekyll makes sense when the site grows to multiple pages or the content is Markdown-
  driven by a non-developer. This page is developer-maintained and content-stable.
- MkDocs is a documentation site generator, not a landing page tool. It outputs
  navigation-heavy multi-page sites. It belongs in `documentation/`, not `docs/`.
- Plain HTML + CSS avoids all of the above: zero build steps, zero Ruby/Node
  dependencies, instant GitHub Pages deploy from the `/docs` folder, no version
  drift between generators.

**File layout under `docs/`**:

```
docs/
├── index.html          — the single page
├── style.css           — all styles (no framework)
├── assets/
│   ├── demo.gif        — screen recording (added Phase 5)
│   ├── pipeline.svg    — architecture diagram
│   └── confidence.svg  — tier explainer diagram
└── fonts/              — self-hosted JetBrains Mono + Inter (optional)
```

No `_config.yml`, no Gemfile, no package.json, no build output directory.
The `/docs` folder is the published root; every path in `index.html` is relative
to that root.

### CSS constraints

- No CSS framework (no Bootstrap, Tailwind, etc.). The page is short enough that a
  bespoke ~200-line CSS file is cleaner and more maintainable.
- Responsive via CSS Grid / Flexbox. Two breakpoints: mobile (< 640px) and desktop
  (≥ 640px). No tablet-specific breakpoint needed for a single-column page.
- Dark theme only — no light-mode toggle. The palette is fixed (see Visual identity).
- No JavaScript unless required for a single specific interaction (e.g., copy-to-
  clipboard on the ledger code block). If JS is used, it is a single inline script
  block; no external JS dependencies, no bundler.

## Content & accessibility rules

### Accessibility baseline (WCAG 2.1 AA)

| Rule | Requirement |
|---|---|
| Colour contrast | All text/background pairs must meet 4.5:1 minimum. Primary text `#e6edf3` on `#0d1117` = 15.2:1 (passes). Accent yellow `#d29922` on `#0d1117` = 5.4:1 (passes). Verify all custom pairings with a contrast checker before publishing. |
| Semantic HTML | Heading hierarchy: one `<h1>` (project name), `<h2>` per section, no skipped levels. |
| Image alt text | Every `<img>` and `<svg>` embedded as image carries a descriptive `alt`. SVGs embedded inline use `<title>` and `aria-labelledby`. |
| Keyboard navigation | All CTAs, links, and interactive elements reachable and operable via keyboard. `focus-visible` style must be visible (not reset to `outline: none`). |
| GIF / video | No auto-play. If the demo is an animated GIF, a static fallback image plus a "pause animation" affordance is required for users with vestibular disorders (CSS `prefers-reduced-motion` media query stops the GIF). |
| Language attribute | `<html lang="en">` required. |
| Link text | No "click here", no bare URLs as link text. Every link has a descriptive label. |

### Responsive rules

- Single-column layout on mobile (< 640px). No horizontal scrolling.
- The architecture diagram SVG scales with `width: 100%; height: auto` — never
  a fixed pixel width that clips on mobile.
- The ledger JSON code block wraps or scrolls horizontally (not clipped) on narrow
  viewports.
- The confidence-tier table collapses to a stacked card layout on mobile (three
  cards, one per tier) when the table would overflow.

### Reproducibility constraint for examples

Every code example and data excerpt on the page must come from an actual run on
real data, not an invented value. Concretely:
- The ledger JSON block comes from a real run (identified by date comment).
- The demo GIF is recorded from a real end-to-end pipeline execution.
- The confidence-tier table examples use values from the Kansas/Schaben run
  (well name and depth range cited in the caption).

This is a content rule enforced by the project ethos: the site makes the same
honesty promise as the system. Invented examples would contradict the traceability
claim in the hero pitch.

### Content update policy

The site is updated at two events only:
1. A new phase of the roadmap completes and delivers a new verifiable artifact
   (e.g., Phase 5 delivers the first prose report → demo section is populated).
2. A factual detail changes (a library version is updated, a metric threshold is
   finalized at Phase 7 exit).

Between those events the site is static. It does not track in-progress work or
reflect the current PLAN.md state. The PLAN.md is the execution record; the site
is the public-facing evidence of what is done.
