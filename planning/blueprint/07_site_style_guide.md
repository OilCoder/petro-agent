# Site Strategy Guide — petro-agent

This document is the **strategy anchor** for the GitHub Pages landing site published
from `docs/`. It governs messaging, positioning, audience, tone, and content rules —
the *what the site says and to whom*. It builds on the Charter's goal statement and
stakeholder list (`00_charter.md`) and is distinct from the report output format
(defined in `03_source_sink_contracts.md`) and the code documentation layer
(generated by `/document` into `documentation/`).

**Visual truth lives elsewhere.** Per the Topic 11 split, the visual specifics
(palette, typography, iconography, page-tech stack, CSS/accessibility implementation)
are owned by an HTML **living style guide** under `docs/` — a self-demonstrating page
that renders the design tokens it defines. A prose doc cannot own visual truth, so it
is not duplicated here. This document owns only the strategy; the `docs/` living style
guide owns the pixels. Build the style scaffold (tokens, fonts, color, type scale,
component shells) now; the full site waits for a rendered report/ledger to showcase
(post-Phase 5).

## Purpose & audience

The `docs/` site is a single-page portfolio landing that communicates the petro-agent
project to three audiences simultaneously:

| Audience | What they need to see | Success signal |
|---|---|---|
| Recruiters / non-technical evaluators | Evidence the system is trustworthy and production-grade, not just impressive-sounding | They reach out or bookmark the repo |
| Potential clients (petrophysicists, operators) | Correct problem framing, defensible methodology, honest ceiling | They ask for a demo or a conversation |
| Technical peers (ML engineers, petrophysicists, collaborators) | Architectural soundness, ledger design, calibration methodology, correct use of domain knowledge | They fork, cite, or open an issue |

The site is not a tutorial, not a product sales page, and not a research paper.
It is a concise, verifiable argument that the system does what it claims, and that
what it claims is worth doing.

## Messaging / positioning

### The one-sentence pitch

> "An autonomous petrophysical report engine: every number from tested deterministic
> code, every claim traceable to its evidence, and confidence calibrated — not assumed."

This sentence must appear in the hero section verbatim (or as a direct paraphrase).
It encodes the three pillars:

1. **Tested deterministic code** — the LLM never computes; it orchestrates.
2. **Every claim traceable** — the JSON ledger ties each output to input curves,
   function version, parameters, and parameter provenance.
3. **Calibrated confidence** — the system assigns `FIRM`, `QUALIFIED`, or `BRACKETED`
   tiers based on parameter provenance; tone is bound to the tier by policy-as-code.

### The honest ceiling statement

A short paragraph under the pitch must state the honest ceiling explicitly:

> "The system does not promise correct results. It promises that confidence is
> calibrated: when parameters are unconstrained by core data, the output says so
> explicitly and states the propagated uncertainty range. Honest about what it does
> not know — that is the design, not a disclaimer."

This language is non-negotiable. The Charter's first Non-goal forbids promising
"always correct" results. The site must not use language that implies it.

### What the site must NOT say

- "AI-powered petrophysics" (implies the LLM computes the numbers — it does not).
- "Accurate interpretation" or "reliable results" without qualification.
- Any implied claim that the system replaces domain-expert judgment for well evaluation.
- Phrases that suggest the LLM is the primary reliability mechanism (the deterministic
  harness is the primary reliability mechanism; the LLM orchestrates and writes prose).

### Positioning in three bullets

The site may use a short three-bullet "what makes this different" block:

1. The engine is deterministic, version-pinned, and golden-tested — not generated by
   the LLM at runtime.
2. The confidence tier is assigned by a deterministic gating stage, not by the writer
   agent — tone cannot outrun the evidence.
3. Every number traces in a JSON ledger to its input curve, function, version,
   parameters, and validator results.

## Tone & voice

### Governing principle

The voice is that of a careful engineer who solved a hard problem honestly. It does
not boast, does not hedge into vagueness, and does not use adjectives where numbers
or mechanisms can speak.

### Positive constraints

- **Technical precision over marketing language.** Name the mechanism: "the
  LangGraph orchestrator routes the objection to the gating stage" is better than
  "our intelligent pipeline handles errors automatically."
- **Concrete over comparative.** "MAE < 0.03 v/v on PHIE against the VOLVE accepted
  interpretation" is better than "state-of-the-art accuracy."
- **First-person singular, sparing.** Acceptable in the author bio / about section.
  Avoid "we" — this is a solo portfolio project.
- **Active voice throughout.** "The ledger records parameter provenance" not "parameter
  provenance is recorded by the ledger."

### Negative constraints

- No exclamation points.
- No "cutting-edge", "revolutionary", "powerful", "smart", or "robust" as adjectives
  unanchored to a mechanism or measurement.
- No "etc." at the end of lists — complete the list or cut it.
- No teaser language ("coming soon") for features not yet built. Open questions in the
  site's context are listed under `## Open questions` in this document, not on the page.

### Calibration-specific language

When describing the confidence system on the page, use the exact tier names from the
system: `FIRM`, `QUALIFIED`, `BRACKETED`. Do not substitute synonyms (e.g., "high
confidence", "medium confidence") — the tier names are part of the technical interface
and their precise meaning is described in the "calibrated-confidence explainer" section.

## Content rules

### Reproducibility constraint for examples

Every code example and data excerpt on the page must come from an actual run on
real data, not an invented value. Concretely:
- The ledger JSON block comes from a real run (identified by date comment).
- The demo GIF is recorded from a real end-to-end pipeline execution.
- The confidence-tier table examples use values from the Kansas/Schaben run
  (well name and depth range cited in the caption).

This is a content rule enforced by the project ethos: the site makes the same
honesty promise as the system. Invented examples would contradict the traceability
claim in the hero pitch.

### Content update policy

The site is updated at two events only:
1. A new phase of the roadmap completes and delivers a new verifiable artifact
   (e.g., Phase 5 delivers the first prose report → demo section is populated).
2. A factual detail changes (a library version is updated, a metric threshold is
   finalized at Phase 7 exit).

Between those events the site is static. It does not track in-progress work or
reflect the current PLAN.md state. The PLAN.md is the execution record; the site
is the public-facing evidence of what is done.
